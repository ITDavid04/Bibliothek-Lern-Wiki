# Das Master-Theorem – einfach erklärt

Stell dir vor, du räumst dein Zimmer auf 🛏️

Dein Zimmer ist riesig. Du denkst dir:

> „Zu viel auf einmal – ich teile es in zwei Hälften!“

Du räumst Hälfte 1 auf, dann Hälfte 2, und am Ende stellst du noch zwei Dinge an ihren Platz (= zusammenführen).

Aber was, wenn jede Hälfte immer noch zu groß ist? Dann teilst du wieder, und wieder, und wieder ...

Genau das macht ein **Divide-and-Conquer-Algorithmus**.

---

## Das eigentliche Problem: Wie lange dauert das?

Wenn ein Programm sich selbst immer wieder aufruft, möchte man wissen:

> „Wie viele Schritte braucht der Computer insgesamt, wenn das Problem *n* Elemente hat?“

Das ist schwer zu berechnen, weil die Aufrufe ineinander verschachtelt sind.

Das **Master-Theorem** ist eine Abkürzung – es gibt dir die Antwort, ohne dass du alles ausrechnen musst.

---

## Die Formel – ohne Panik! 😅

```
T(n) = a · T(n/b) + f(n)
```

Klingt wild. Ist es aber nicht:

| Symbol | Bedeutung |
|--------|-----------|
| `T(n)` | Wie lange brauche ich für *n* Elemente? |
| `a`    | In wie viele **gleich große** Teile zerlegst du das Problem? |
| `T(n/b)` | Jedes Teilproblem hat die Größe **n / b** (also *b*-mal kleiner) |
| `f(n)` | Wie viel Arbeit kostet das **Zerlegen + Zusammenbauen**? |

---

## Konkretes Beispiel: Telefonbuch durchsuchen 📖

Du suchst „Müller“ in einem Telefonbuch mit 1000 Seiten:

1. Schlage die Mitte auf.
2. Ist Müller davor oder danach?
3. Wirf die andere Hälfte weg.
4. Wiederhole mit der restlichen Hälfte.

Das ergibt:

```
T(n) = 1 · T(n/2) + O(1)
```

- `a = 1` → du schaust nur in **1** Teilproblem weiter  
- `b = 2` → das Problem wird **halbiert**  
- `f(n) = O(1)` → das Vergleichen kostet immer **gleich viel** (konstant)

---

## Der entscheidende Vergleich

Das Master-Theorem stellt eine simple Frage:

> **Was kostet mehr – die Rekursion oder das Zusammenbauen?**

Stell dir einen Baum vor:

```
                    [ganzes Problem]
                   /                \
          [linke Hälfte]      [rechte Hälfte]
          /          \          /           \
      [Viertel]  [Viertel]  [Viertel]   [Viertel]
         ...        ...        ...          ...
```

Jede Ebene kostet etwas Arbeit. Die Frage ist nur: **Welche Ebene dominiert?**

---

## Die 3 Fälle – ganz simpel

### Fall 1 – Die vielen Blätter sind das Problem 🌿

- Die **unterste Ebene** (die vielen winzigen Teilprobleme) kostet am meisten.  
- Das Zusammenbauen ist vergleichsweise billig.

**Beispiel:** Du hast 7 Teilprobleme, aber das Zusammenbauen ist günstig.

```
⇒ Laufzeit = Θ(n^(log_b a))
```

> Merke: Die Masse der Mini-Aufgaben ganz unten bestimmt die Zeit.

---

### Fall 2 – Alles gleich teuer ⚖️

- **Jede Ebene** des Baumes kostet gleich viel Arbeit.  
- Es gibt etwa `log_b n` Ebenen, also multipliziert sich die Arbeit pro Ebene mit der Anzahl der Ebenen.

**Beispiel:** Mergesort – du halbierst immer, und das Zusammenführen kostet genauso viel wie die ganze Ebene.

```
⇒ Laufzeit = Θ(n^(log_b a) · log n)
```

> Merke: Der zusätzliche `log n`-Faktor zählt einfach, wie oft du geteilt hast.

---

### Fall 3 – Das Zusammenbauen ist das Problem 🔧

- Der **erste Aufruf ganz oben** (die Wurzel) kostet so viel, dass der Rest kaum ins Gewicht fällt.  
- Voraussetzung: Die Arbeit wird in den tieferen Ebenen **wirklich** viel kleiner (bei normalen Funktionen wie `n²` oder `n³` automatisch erfüllt).

```
⇒ Laufzeit = Θ(f(n))
```

> Merke: Das oberste „Zerlegen + Zusammenbauen“ allein gibt den Ton an.

---

## Zurück zu den Beispielen

| Algorithmus   | Was passiert?                          | Ergebnis       |
|---------------|----------------------------------------|----------------|
| **Binärsuche** | Halbieren, kaum Zusammenbau-Aufwand    | `O(log n)`     |
| **Mergesort**  | Halbieren + Zusammenführen gleich teuer| `O(n log n)`   |
| **Strassen**   | 7 Teilprobleme, Zusammenführen billig  | `O(n^{2,81})`  |

## Die wichtigste Intuition 💡

```
┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  Das Master-Theorem schaut nur, WER die meiste Arbeit macht: │
│                                                               │
│  Die vielen kleinen     Alle gleich      Das große Problem    │
│  Teilprobleme unten?    teuer?           ganz oben?           │
│         │                  │                   │              │
│       Fall 1             Fall 2             Fall 3            │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

Du musst nicht den ganzen Rekursionsbaum ausrechnen.  
Du vergleichst nur zwei Dinge miteinander – und schaust, wer gewinnt. Das war’s. 🎯

## Glossar

| Begriff | Erklärung |
|---|---|
| **Divide-and-Conquer** | Algorithmusprinzip: Teile das Problem, löse Teile rekursiv, füge Teillösungen zusammen |
| **Rekurrenzgleichung** | Gleichung, die `T(n)` durch `T` kleinerer Argumente ausdrückt |
| **Asymptotische Analyse** | Beschreibung des Wachstums einer Funktion für große `n` (O, Ω, Θ) |
| **Θ (Theta)** | Exaktes asymptotisches Wachstum – weder nur oben noch nur unten begrenzt |
| **Rekursionsbaum** | Visualisierung aller rekursiven Aufrufe als Baum; Blätter = Basisfälle |
| **Regularitätsbedingung** | Technische Voraussetzung für Fall 3: `a·f(n/b) ≤ c·f(n)` für `c < 1` |
| **Blattarbeit** | Gesamtaufwand aller Basisfälle: `n^(log_b a)` |
| **Akra-Bazzi-Theorem** | Verallgemeinerung des Master-Theorems für ungleich große Teilprobleme |
| **log_b a** | "Logarithmus von a zur Basis b" – der kritische Exponent im Master-Theorem |
