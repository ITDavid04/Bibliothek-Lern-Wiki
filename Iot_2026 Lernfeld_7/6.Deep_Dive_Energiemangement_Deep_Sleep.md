## 1️⃣ 🔬 Die Physik dahinter: Warum Energie sparen?

### **Das Grundproblem**

```

Batterie (3,7V / 2000mAh) mit ESP32 im Dauerbetrieb:
├── Dauerbetrieb (240 MHz, WiFi aktiv): ~200mA → 10 Stunden
├── Light Sleep (80 MHz, WiFi pausiert): ~2mA  → 41 Tage
└── Deep Sleep (nur RTC aktiv):         ~10µA  → 27 Jahre (theoretisch!)
```

> **IHK-Merksatz:** *"Deep Sleep ist kein Ruhezustand – es ist ein kontrollierter Neustart mit Gedächtnis."*

### **Die Energieformel (IHK-relevant)**

```

Batterielaufzeit (h) = Kapazität (mAh) / Durchschnittsverbrauch (mA)

Beispiel ESP32 (Deep Sleep 23h/Tag, Aktiv 1h/Tag):
├── Aktiv:      1h  × 200mA  =  200.00 mAh
├── Deep Sleep: 23h × 0.01mA =    0.23 mAh
├── Gesamt:                     200.23 mAh/Tag
└── Laufzeit:   2000mAh / 200.23mAh ≈ 10 Tage
```

### **🔍 IHK-Prüfungsfragen**

✅ **Frage:** *"Ein IoT-Sensor soll 1 Jahr mit einer 2000mAh-Batterie laufen. Welche Maßnahmen sind notwendig?"*

✔ **Antwort:**

1. **Deep Sleep** als primären Betriebsmodus nutzen
2. **Messintervall optimieren** (z. B. alle 15 Minuten statt jede Minute)
3. **QoS 0** für MQTT (kein Handshake-Overhead – nur für unkritische Daten geeignet, da keine Zustellgarantie!)
4. **Sendeleistung reduzieren** (WiFi TX-Power von 20dBm auf 11dBm)
5. **Peripherie abschalten** (Sensoren über MOSFET nur bei Bedarf einschalten)

## 2️⃣ 💤 Sleep-Modi im Detail: Was passiert wirklich?

### **ESP32 Sleep-Modi – Vergleichstabelle**

| **Modus** | **CPU** | **RAM** | **WiFi/BT** | **RTC** | **Strom** | **Aufwachzeit** |
|-----------|---------|---------|-------------|---------|-----------|-----------------|
| **Aktiv** | ✅ An | ✅ An | ✅ An | ✅ An | ~200mA | — |
| **Modem Sleep** | ✅ An | ✅ An | 💤 Pausiert | ✅ An | ~20mA | <1ms |
| **Light Sleep** | 💤 Pausiert | ✅ An | 💤 Pausiert | ✅ An | ~2mA | ~2ms |
| **Deep Sleep** | ❌ Aus | ❌ Aus | ❌ Aus | ✅ An | ~10µA | ~300ms |
| **Hibernation** | ❌ Aus | ❌ Aus | ❌ Aus | ⚠️ Minimal | ~5µA | ~300ms |

> **Wichtig für FIAE:** Deep Sleep = RAM-Verlust! Variablen müssen in **RTC-Memory** oder **Flash** gespeichert werden.

### **Was überlebt den Deep Sleep?**

```

Deep Sleep Übersicht:
│
├── ✅ ÜBERLEBT
│   ├── RTC Memory (machine.RTC().memory()) – 8KB
│   ├── RTC_SLOW_MEM – für ULP-Programm
│   └── Flash-Speicher (NVS, Dateisystem)
│
└── ❌ VERLOREN
    ├── Alle normalen RAM-Variablen
    ├── WiFi-Verbindung (muss neu aufgebaut werden)
    └── Alle laufenden Prozesse
```

### **Aufwach-Quellen & Reset-Ursachen (IHK-relevant!)**

| **Wake-up Quelle** | **machine.reset_cause()** | **Typischer Einsatz** |
|--------------------|--------------------------|----------------------|
| Timer | `machine.DEEPSLEEP_RESET` | Regelmäßige Messungen |
| EXT0 (einzelner GPIO) | `machine.EXT0_WAKE` | Türsensor, Taster |
| EXT1 (mehrere GPIOs) | `machine.EXT1_WAKE` | Mehrere Sensoren |
| Touch-Pad | `machine.TOUCHPAD_WAKE` | Wearables |
| Power-On / Reset | `machine.PWRON_RESET` | Erststart |

> ⚠️ **Kritisches Detail:** `DEEPSLEEP_RESET` tritt **nur beim Timer-Wake-up** auf.  
> Externer GPIO-Wake-up liefert `EXT0_WAKE` oder `EXT1_WAKE` – **nicht** `DEEPSLEEP_RESET`!  
> Dieser Unterschied ist prüfungsrelevant und ein häufiger Programmierfehler.

## 3️⃣ 💻 Code: Deep Sleep Patterns (FIAE)

### **Pattern 1: Timer Wake-up (Standard)**

```python
# ESP32 MicroPython: Timer Wake-up Pattern
# Installation: import upip; upip.install('micropython-umqtt.simple')

import machine
import network
import time
from umqtt.simple import MQTTClient
from machine import I2C, Pin
import bme280

# ─── Konfiguration ────────────────────────────────────────────
WIFI_SSID      = "dein_SSID"
WIFI_PASS      = "dein_Passwort"
MQTT_BROKER    = "test.mosquitto.org"
MQTT_TOPIC     = b"sensor/wohnzimmer/temperature"
SLEEP_TIME_MS  = 5 * 60 * 1000  # 5 Minuten

# ─── Sensor auslesen ──────────────────────────────────────────
def read_sensor():
    i2c = I2C(0, scl=Pin(22), sda=Pin(21), freq=400000)
    bme = bme280.BME280(i2c=i2c)
    bme.read_compensated_data()  # PFLICHT: erst lesen, dann abrufen!
    return {
        "temp":     bme.temperature,
        "humidity": bme.humidity,
        "pressure": bme.pressure
    }

# ─── WiFi verbinden ───────────────────────────────────────────
def connect_wifi():
    wlan = network.WLAN(network.STA_IF)
    wlan.active(True)
    if not wlan.isconnected():
        wlan.connect(WIFI_SSID, WIFI_PASS)
        timeout = 10
        while not wlan.isconnected() and timeout > 0:
            time.sleep(1)
            timeout -= 1
    if not wlan.isconnected():
        raise RuntimeError("WiFi-Verbindung fehlgeschlagen!")

# ─── MQTT senden ──────────────────────────────────────────────
def publish_data(data):
    client = MQTTClient("esp32_timer", MQTT_BROKER, keepalive=30)
    client.connect()
    import ujson
    client.publish(MQTT_TOPIC, ujson.dumps(data), qos=0)
    client.disconnect()  # Saubere Trennung → kein LWT ausgelöst

# ─── Hauptprogramm ────────────────────────────────────────────
def main():
    try:
        data = read_sensor()
        connect_wifi()
        publish_data(data)
        print(f"Gesendet: {data}")
    except Exception as e:
        print(f"Fehler: {e}")
    finally:
        print(f"Schlafe {SLEEP_TIME_MS // 60000} Minuten...")
        machine.deepsleep(SLEEP_TIME_MS)  # MicroPython-Syntax!

main()
```

### **Pattern 2: Deep Sleep mit Zählervariable (RTC Memory)**

```python
# ESP32 MicroPython: RTC Memory Pattern
# Zählt Aufwachzyklen persistent über Deep Sleep hinweg

import machine
# Hinweis: 'import esp32' hier entfernt – wird in diesem Pattern nicht benötigt

rtc = machine.RTC()

def get_boot_count():
    """Lese Aufwachzähler aus RTC Memory (überlebt Deep Sleep!)."""
    try:
        data = rtc.memory()
        if len(data) >= 4:
            return int.from_bytes(data[:4], 'little')
    except:
        pass
    return 0

def save_boot_count(count):
    """Speichere Aufwachzähler in RTC Memory."""
    rtc.memory(count.to_bytes(4, 'little'))

# ─── Aufwachgrund ermitteln ───────────────────────────────────
def get_wakeup_reason():
    reasons = {
        machine.PWRON_RESET:     "Neustart (Power-On)",
        machine.DEEPSLEEP_RESET: "Deep Sleep – Timer",     # NUR Timer!
        machine.EXT0_WAKE:       "Externer Pin (EXT0)",
        machine.EXT1_WAKE:       "Externer Pin (EXT1)",
        machine.TOUCHPAD_WAKE:   "Touch-Pad",
        machine.ULP_WAKE:        "ULP-Coprozessor",
    }
    return reasons.get(machine.reset_cause(), "Unbekannt")

# ─── Hauptprogramm ────────────────────────────────────────────
boot_count = get_boot_count() + 1
save_boot_count(boot_count)

print(f"Aufwachzyklus #{boot_count}")
print(f"Aufwachgrund:  {get_wakeup_reason()}")

if boot_count % 10 == 0:
    print("Sende Diagnose-Daten...")
    # ... MQTT publish ...

machine.deepsleep(60 * 1000)  # 1 Minute
```

### **Pattern 3: GPIO Wake-up (Türsensor)**

> ✅ **v1.1 Korrektur:** `DEEPSLEEP_RESET` → `EXT0_WAKE` (kritischer Bug behoben)

```python
# ESP32 MicroPython: EXT0 GPIO Wake-up
# Use Case: Türsensor – Aufwachen wenn Tür geöffnet wird

import machine
import esp32
from machine import Pin

DOOR_PIN   = 33   # Muss RTC-fähiger Pin sein!
MQTT_TOPIC = b"sensor/tuer/status"

# ─── Aufwachgrund prüfen ──────────────────────────────────────
reset_cause = machine.reset_cause()

if reset_cause == machine.PWRON_RESET:
    print("Erststart – initialisiere System...")

elif reset_cause == machine.EXT0_WAKE:
    # ✅ KORREKT: EXT0_WAKE, NICHT DEEPSLEEP_RESET!
    # DEEPSLEEP_RESET tritt nur beim Timer-Wake-up auf.
    # Externer GPIO-Wake-up liefert immer EXT0_WAKE.
    print("Tür geöffnet! Sende Alarm...")

    # Hier: WiFi verbinden + MQTT publish (siehe Pattern 1)
    # client.disconnect() am Ende → kein LWT ausgelöst

# ─── EXT0 Wake-up konfigurieren ───────────────────────────────
# WICHTIG: Nur RTC-fähige GPIOs! (0,2,4,12-15,25-27,32-39)
# GPIO 34-39: Nur INPUT | GPIO 12: Beeinflusst Boot-Modus!
door_pin = Pin(DOOR_PIN, Pin.IN, Pin.PULL_UP)

esp32.wake_on_ext0(
    pin=door_pin,
    level=esp32.WAKEUP_ALL_LOW   # Aufwachen bei LOW (Kontakt geschlossen)
)

print("Warte auf Tür-Event... (Deep Sleep)")
machine.deepsleep()  # Ohne Timeout = schläft bis GPIO-Event
```

### **Pattern 4: Peripherie abschalten (MOSFET-Trick)**

> ✅ **v1.1 Korrektur:** GPIO-Initialwert `value=1` verhindert Floating nach Deep Sleep

```python
# ESP32 MicroPython: Peripherie-Steuerung via MOSFET
# Use Case: BME280 läuft dauerhaft mit ~1.8mA – unnötig!

import machine
import time
from machine import Pin, I2C
import bme280

# ─── Hardware-Setup ───────────────────────────────────────────
# P-Channel MOSFET Gate an GPIO 26
# LOW  = MOSFET leitend  = Sensor AN
# HIGH = MOSFET gesperrt = Sensor AUS
#
# ✅ value=1 beim Init: GPIO startet nach Deep Sleep sofort HIGH
# → MOSFET bleibt gesperrt bis sensor_on() explizit aufgerufen wird
# → Verhindert kurzzeitigen ungewollten Stromfluss durch floatenden Pin
SENSOR_POWER_PIN = Pin(26, Pin.OUT, value=1)  # Sicher AUS beim Start!

def sensor_on():
    """Sensor einschalten und Stabilisierungszeit abwarten."""
    SENSOR_POWER_PIN.value(0)   # P-Channel: LOW = AN
    time.sleep_ms(50)            # Warten bis Sensor stabil und I2C bereit

def sensor_off():
    """Sensor ausschalten."""
    SENSOR_POWER_PIN.value(1)   # P-Channel: HIGH = AUS

# ─── Messung mit Power-Management ─────────────────────────────
def measure_with_powerdown():
    sensor_on()
    try:
        i2c = I2C(0, scl=Pin(22), sda=Pin(21))
        bme = bme280.BME280(i2c=i2c)
        bme.read_compensated_data()  # PFLICHT vor Datenzugriff!
        return {
            "temp":     bme.temperature,
            "humidity": bme.humidity
        }
    finally:
        sensor_off()  # IMMER ausschalten – auch bei Exception!

# ─── Hauptprogramm ────────────────────────────────────────────
data = measure_with_powerdown()
print(f"Messung: {data}")
# ... MQTT publish ...
machine.deepsleep(10 * 60 * 1000)  # 10 Minuten
```

## 4️⃣ 🔌 FISI: Hardware & Konfiguration

### **ESP32 Pin-Referenz (Deep Sleep relevant)**

```
ESP32 RTC-fähige GPIOs (für EXT0/EXT1 Wake-up):
┌─────────────────────────────────────────────────┐
│  ✅  Geeignet:  0, 2, 4, 12, 13, 14, 15,        │
│                25, 26, 27, 32, 33               │
│  ⚠️  Input-only (kein OUTPUT): 34, 35, 36, 39   │
│  ❌  GPIO 12:   Beeinflusst Boot-Modus!          │
│  ✅  Empfehlung: GPIO 33 für externen Sensor     │
└─────────────────────────────────────────────────┘
```

### **Typische Stromspar-Schaltung**


```
          3.3V
           │
          [R1 10kΩ Pull-up]
           │
ESP32 GPIO26 ──[Gate]
                  │
           P-MOSFET (z.B. AO3401)
           [Source]──── 3.3V
           [Drain] ──── VCC Sensor (BME280)
                              │
                           [BME280]
                              │
                            GND
```

### **🛠️ Mosquitto-Konfiguration für Low-Power Clients**

```ini
# /etc/mosquitto/mosquitto.conf
# Optimiert für Deep-Sleep-Clients

listener 1883
protocol mqtt

# Keep-Alive Toleranz: mind. 2× das Sleep-Intervall
max_keepalive 600

# Persistent Sessions: Nachrichten für QoS 1/2 puffern
# (Nur relevant wenn Client mit cleanSession=False verbindet)
persistence true
persistence_location /var/lib/mosquitto/

log_type all
log_dest file /var/log/mosquitto/mosquitto.log
```

> **LWT-Hinweis:** LWT (Last Will) wird nur ausgelöst, wenn der Client **unerwartet** die Verbindung verliert.  
> Deep-Sleep-Clients trennen sich sauber mit `client.disconnect()` (siehe Pattern 1) – **LWT wird dabei nicht ausgelöst**.  
> LWT ist sinnvoll bei Absturz oder Stromausfall, nicht als reguläres Offline-Signal für Sleep-Clients.

### **🔍 IHK-Prüfungsfragen (FISI)**

✅ **Frage:** *"Ein ESP32 sendet alle 10 Minuten Daten. Welche Mosquitto-Einstellung ist wichtig?"*

✔ **Antwort:**

- **`max_keepalive`** auf mindestens 600 Sekunden setzen
- **Persistent Sessions** aktivieren (Nachrichten für QoS 1/2 werden gepuffert)
- **LWT** nur für Absturz-Erkennung sinnvoll – bei sauberem `disconnect()` wird es **nicht** ausgelöst

## 5️⃣ 📊 Energieverbrauch messen & berechnen

### **Laufzeit-Kalkulator**

```python
# Energieverbrauch-Kalkulator (direkt für IHK-Aufgaben verwendbar)

def calculate_battery_life(
    capacity_mah:      float,   # Batteriekapazität in mAh
    active_current_ma: float,   # Strom im Aktiv-Modus in mA
    active_time_s:     float,   # Aktiv-Zeit pro Zyklus in Sekunden
    sleep_current_ua:  float,   # Strom im Deep Sleep in µA
    sleep_time_s:      float,   # Sleep-Zeit pro Zyklus in Sekunden
    efficiency:        float = 0.85  # LDO-Regler ~85% Wirkungsgrad
) -> dict:

    active_charge_mah = (active_current_ma * active_time_s) / 3600
    sleep_charge_mah  = (sleep_current_ua / 1000 * sleep_time_s) / 3600
    total_charge_mah  = (active_charge_mah + sleep_charge_mah) / efficiency

    cycle_time_h      = (active_time_s + sleep_time_s) / 3600
    avg_current_ma    = total_charge_mah / cycle_time_h

    battery_life_h    = capacity_mah / avg_current_ma

    return {
        "avg_current_ma":     round(avg_current_ma, 4),
        "battery_life_hours": round(battery_life_h, 1),
        "battery_life_days":  round(battery_life_h / 24, 1),
        "battery_life_years": round(battery_life_h / 24 / 365, 2)
    }

# ─── IHK-Beispielaufgabe ──────────────────────────────────────
result = calculate_battery_life(
    capacity_mah      = 2000,
    active_current_ma = 180,
    active_time_s     = 5,
    sleep_current_ua  = 10,
    sleep_time_s      = 600,
    efficiency        = 0.85
)

print(f"Ø-Strom:   {result['avg_current_ma']} mA")
print(f"Laufzeit:  {result['battery_life_days']} Tage")
print(f"           {result['battery_life_years']} Jahre")
```

### **🔍 IHK-Rechenweg (klausurfertig)**

✅ **Frage:** *"ESP32, 2000mAh, 200mA aktiv für 3s alle 15 Minuten, 10µA Deep Sleep"*

```
Aktiv:       200mA   × (3/3600)h       = 0.1667 mAh/Zyklus
Deep Sleep:  0.01mA  × (897/3600)h     = 0.0025 mAh/Zyklus
Summe:       (0.1667 + 0.0025) / 0.85  = 0.1991 mAh/Zyklus

Zyklen/h:    60min / 15min             = 4
Ø-Strom:     0.1991 × 4               = 0.796 mA

Laufzeit:    2000mAh / 0.796mA         = 2512h ≈ 104 Tage
```
