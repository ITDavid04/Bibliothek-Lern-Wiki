---

## 1. 🗺️ Die drei Schnittstellen im Überblick

Mikrocontroller und Einplatinencomputer (z. B. Raspberry Pi, ESP32, STM32) kommunizieren mit angeschlossenen Chips (Sensoren, Krypto-Elementen, Displays) über standardisierte Bus-Protokolle.

### 🔄 Vergleichstabelle

| Eigenschaft | **I2C** | **SPI** | **GPIO** |
|---|---|---|---|
| **Leitungen** | 2 (SDA + SCL) | 4 (MOSI, MISO, SCK, CS) | 1 pro Pin |
| **Geschwindigkeit** | Langsam–mittel (100–400 kHz) | Schnell (bis 50 MHz+) | Kein Protokoll |
| **Mehrere Geräte** | Ja, über Adresse (7-Bit) | Ja, über eigenen CS-Pin | – |
| **Typische Anwendung** | Sensoren, EEPROMs, TPMs | Flash-Speicher, Displays, SD-Karten | Taster, LEDs, UART Debug |
| **Authentifizierung** | ❌ Nativ keine | ❌ Nativ keine | ❌ Nativ keine |

> **Kernaussage für die Prüfung:** Keines dieser Protokolle bringt von Haus aus Authentifizierung oder Verschlüsselung mit. Sicherheit muss auf höheren Ebenen (Software, Krypto-Chips) implementiert werden.

## 2. 🔗 Relevanz für den Hardware Root of Trust

Im Hauptdokument wurde der **Hardware Root of Trust** (TPM 2.0, Secure Element) vorgestellt. Diese Chips sind physisch über I2C oder SPI an den Hauptprozessor angebunden – das ist der entscheidende Zusammenhang:

```
[Hauptprozessor (MCU)]
        │
        ├── I2C-Bus ──> [TPM 2.0 / Krypto-Element]
        │                   └─ Privater Schlüssel (verlässt den Chip nie)
        │
        └── SPI-Bus ──> [Flash-Speicher]
                            └─ Firmware-Image (Bank A / Bank B)
```

* Der TPM antwortet auf Krypto-Anfragen des Prozessors **über den I2C-Bus**.
* Der Flash-Speicher (mit den OTA-Banks) ist über **SPI** direkt auslesbar – wenn kein Schutz aktiv ist.

## 3. ⚠️ Physische Angriffsvektoren

### 🔹 SPI-Flash Direktauslesen

Der häufigste Angriff auf eingebettete Systeme in der Praxis:

```
Angreifer mit Logikanalysator / Clip
        │
        └──> Klemmt direkt an SPI-Datenleitungen des Flash-Chips
                        │
                        └──> Liest gesamten Flash-Inhalt aus:
                             - Firmware-Binary
                             - Konfigurationsdaten
                             - Ggf. Schlüsselmaterial (wenn nicht im TPM!)
```

**Gegenmaßnahme:** Flash Read-Protection (RDP) auf dem Mikrocontroller aktivieren – sperrt den Lese-Zugriff über SPI auf Hardware-Ebene. Bei STM32-Chips z. B. über Option Bytes konfigurierbar.

---

### 🔹 I2C Bus-Sniffing

I2C ist ein geteilter Bus. Mit einem Logic Analyzer (Hardware ab ca. 5–10 €) lässt sich der gesamte Datenverkehr mitschneiden:

```
[MCU] ──SDA/SCL──> [Sensor / TPM]
           │
           └──> [Logic Analyzer des Angreifers] passt einfach dazwischen
```

**Konsequenz:** Übertragene Messwerte, Adressen und Befehle sind im Klartext lesbar.

**Gegenmaßnahme:** Für Krypto-Operationen immer einen dedizierten Sicherheitschip (Secure Element wie ATECC608) nutzen, der nur Ergebnisse zurückliefert – nie Schlüsselmaterial über den Bus schickt.

---

### 🔹 JTAG / Debug-Pins als Einfallstor

JTAG (Joint Test Action Group) ist eine Debug-Schnittstelle, die in der Produktion häufig offen gelassen wird.

| Risiko | Beschreibung |
|---|---|
| **Speicher-Dump** | Über JTAG kann der komplette Flash- und RAM-Inhalt ausgelesen werden |
| **Breakpoints setzen** | Angreifer können den Programmfluss anhalten und Variablen auslesen |
| **Firmware überschreiben** | Direktes Flashen neuer (manipulierter) Firmware ohne Signaturprüfung |
| **Secure Boot umgehen** | In manchen Implementierungen kann Secure Boot über JTAG deaktiviert werden |

**Gegenmaßnahme (Produktions-Checkliste):**
- JTAG-Schnittstelle nach der Produktion per Fuse deaktivieren (OTP-Bits)
- Physische JTAG-Pads auf der Platine entfernen oder mit Epoxidharz versiegeln
- Nicht benötigte Debug-UART-Ports (RX/TX auf GPIO-Pins) deaktivieren

---

### 🔹 GPIO als versteckter UART

Viele Entwickler lassen UART-Debug-Ausgaben auf GPIO-Pins aktiv – auch in Produktionsgeräten:

```bash
# Typische Ausgabe auf UART0/GPIO wenn kein Lock aktiv ist:
[BOOT] Loading firmware v1.1 from Bank B...
[CRYPTO] Public key hash: a3f9...
[WIFI] Connecting to SSID: Fabriknetz_AP
[WIFI] PSK: geheimes_passwort_123   <── Klartext im Log!
```

**Gegenmaßnahme:** Debug-Logging vor Produktions-Build deaktivieren (`#define NDEBUG` oder Build-Flag), UART-Pins als Input ohne Pull-up konfigurieren.

## 4. 🛡️ Die Schutzpyramide: Von Software bis Hardware

```
        ╔══════════════════════╗
        ║   Applikationsebene  ║  ← TLS, Signaturprüfung, Anti-Rollback
        ╠══════════════════════╣
        ║    Bootloader-Ebene  ║  ← Secure Boot, A/B-System, Watchdog
        ╠══════════════════════╣
        ║    OS / Firmware     ║  ← Debug-Logging aus, UART deaktiviert
        ╠══════════════════════╣
        ║   Hardware-Ebene     ║  ← Flash RDP, JTAG-Lock, Secure Element
        ╚══════════════════════╝
           Physische Ebene       ← Gehäuse, Verguss, Tamper-Detection
```

> **IHK-Merksatz:** Sicherheit ist keine einzelne Maßnahme, sondern ein **Schichtenmodell** – "Defense in Depth". Jede Ebene muss unabhängig von den anderen funktionieren.

### 🔹 Physische Ebene: Vergussmasse (Potting)

Die unterste Schutzschicht ist buchstäblich greifbar: Das Gerätinnere wird mit einer aushärtenden Kunstharzmasse (Epoxid oder Polyurethan) vollständig ausgegossen.

| Maßnahme | Wirkung |
|---|---|
| **Vergussmasse (Potting)** | Platine und Chips vollständig eingebettet – mechanisches Auslöten von Flash-Chips oder Anlöten von Mess-Clips an SPI/I2C-Leitungen wird praktisch unmöglich |
| **Tamper-Detection** | Spezialisierte Chips (z. B. Maxim DS3232) erkennen Öffnungsversuche und löschen automatisch den Schlüsselspeicher |
| **Epoxid auf JTAG-Pads** | Zielgerichtetes Versiegeln der Debug-Pads verhindert physisches Anlöten von Verbindungen |

> **Prüfungsrelevanz:** Vergussmasse ist kein Ersatz für kryptographische Schutzmaßnahmen – sie erhöht lediglich den physischen Aufwand für einen Angreifer (sog. *Tamper Resistance*). Ein entschlossener Angreifer mit Lösemitteln und Zeit kann Verguss ggf. entfernen. Daher gilt: physischer Schutz + kryptographischer Schutz = Defense in Depth.

## 5. 🛠️ FISI vs. FIAE: Praxis-Perspektiven

### 🛠️ Für **Systemintegratoren (FISI)**: Produktions-Härtung

Ein FISI stellt sicher, dass Geräte vor der Auslieferung physisch gehärtet sind:

```bash
# Beispiel: STM32 Flash Read-Protection Level 1 setzen (über st-flash / OpenOCD)
# RDP Level 1: Debugger-Zugriff gesperrt, Flash nicht auslesbar
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg \
  -c "init; stm32f4x option_write_bank 0 0x1FFFC000 0x0 0xFF00AA55; reset; exit"

# JTAG komplett deaktivieren (Beispiel ESP32 via esptool)
esptool.py --chip esp32 burn_efuse JTAG_DISABLE
```

**Checkliste vor Produkt-Auslieferung:**
- [ ] JTAG/SWD-Schnittstelle gesperrt (Fuse/eFuse)
- [ ] Flash Read-Protection aktiv
- [ ] Debug-UART deaktiviert
- [ ] Alle Standard-Passwörter geändert (CRA-Pflicht!)
- [ ] Seriennummer & Zertifikat pro Gerät provisioniert
- [ ] Vergussmasse / Epoxid auf JTAG-Pads (je nach Schutzanforderung)

### 💻 Für **Anwendungsentwickler (FIAE)**: Sicherer I2C-Zugriff auf Krypto-Element

> ⚠️ **Wichtiger Hinweis zur Praxis:** Das folgende Beispiel illustriert das **Konzept** des I2C-Zugriffs auf ein Secure Element. Das ATECC608 verwendet ein proprietäres Kommunikationsprotokoll (CRC-geschützte Pakete, spezifische Wake/Sleep-Sequenzen), das **nicht** direkt über `smbus2` angesprochen werden kann. Für echte Projekte muss die offizielle **Microchip CryptoAuthLib** verwendet werden (C-Bibliothek, Python-Binding: `cryptoauthlib`). Quelle: *Microchip ATECC608 Datasheet DS40002239* / *CryptoAuthLib GitHub Repository*.

```python
import smbus2  # Python-Bibliothek für I2C auf Linux (z. B. Raspberry Pi)
# ACHTUNG: Für produktiven Einsatz mit ATECC608 stattdessen verwenden:
# pip install cryptoauthlib  →  from cryptoauthlib import atcab_sign()

# I2C-Adresse des ATECC608 Secure Element (Standard: 0x60, je nach
# Adresskonfiguration des Chips abweichend – siehe Datenblatt Kapitel 2.2)
SECURE_ELEMENT_ADDR = 0x60
bus = smbus2.SMBus(1)  # I2C-Bus 1

def request_signature_from_secure_element(data_hash: bytes) -> bytes:
    """
    Konzeptuelle Illustration: ECDSA-Signatur vom Secure Element anfordern.
    WICHTIG: Der Private Key verlässt das Element nie – nur die Signatur kommt zurück.
    
    In der Praxis: atcab_sign(key_id, digest, signature) aus cryptoauthlib nutzen.
    Quelle: Microchip ATECC608 Datasheet DS40002239, Kapitel 8 (Sign Command)
    """
    # Wake-up Sequenz senden (gerätespezifisch, vereinfacht dargestellt)
    bus.write_byte(SECURE_ELEMENT_ADDR, 0x00)
    
    # Hash-Daten zur Signierung übermitteln
    # (vereinfacht – echte Implementierung benötigt CRC-Framing der ATCA-Bibliothek)
    bus.write_i2c_block_data(SECURE_ELEMENT_ADDR, 0x41, list(data_hash[:32]))
    
    # Signatur abrufen (64 Bytes für ECDSA P-256)
    signature = bus.read_i2c_block_data(SECURE_ELEMENT_ADDR, 0x00, 64)
    return bytes(signature)

# Verwendung im OTA-Kontext:
firmware_hash = b'\xde\xad\xbe\xef' * 8  # SHA-256 der Firmware (32 Bytes)
sig = request_signature_from_secure_element(firmware_hash)
print(f"Signatur vom Secure Element: {sig.hex()[:16]}...")
# Der Private Key war zu keinem Zeitpunkt im RAM des Hauptprozessors!
```

## 📋 IHK-Schnell-Checkliste

* ✅ **I2C / SPI:** Keine native Authentifizierung – Sicherheit liegt beim angebundenen Chip (Secure Element)
* ✅ **SPI-Flash:** Ohne Read-Protection (RDP) direkt auslesbar – muss in Produktion gesperrt sein
* ✅ **JTAG:** Klassisches Einfallstor – nach Produktion deaktivieren (eFuse/OTP)
* ✅ **GPIO/UART:** Debug-Logs in Produktion immer deaktivieren – Klartextdaten sind ein Sicherheitsrisiko
* ✅ **Defense in Depth:** Kein einzelner Mechanismus reicht – Schutz muss auf jeder Ebene greifen
* ✅ **Secure Element Prinzip:** Krypto-Operationen *im* Chip – Schlüssel verlässt das Element niemals
* ✅ **Vergussmasse (Potting):** Erhöht physischen Aufwand für Angreifer – kein Ersatz für Kryptographie, aber sinnvolle letzte Schicht
