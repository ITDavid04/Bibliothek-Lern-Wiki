## 2. Detaillierte Analyse: LoRaWAN, 5G, MQTT

### 📡 LoRaWAN – König der Langstrecke

**Vorteile:**

- Extrem energieeffizient (Batterielaufzeit >5 Jahre möglich)
- Hohe Reichweite (bis 15 km auf dem Land)
- Geringe Kosten (keine SIM-Karte nötig)

**Nachteile:**

- Sehr geringe Datenrate (nur kleine Datenpakete)
- Keine Echtzeit (Latenz Sekunden bis Minuten)

**Einsatzgebiete:**

- Landwirtschaft 4.0 (Bodenfeuchte, Wetter)
- Smart City (Füllstand von Mülltonnen)
- Logistik (GPS-Tracking)

**Code-Beispiel (LoRaWAN mit LMIC-Bibliothek, OTAA):**

```cpp
#include <lmic.h>
#include <hal/hal.h>

// LoRaWAN-Keys (OTAA) – Beispielwerte
static const u1_t PROGMEM APPEUI[8] = { 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00 };
static const u1_t PROGMEM DEVEUI[8] = { 0x01, 0x02, 0x03, 0x04, 0x05, 0x06, 0x07, 0x08 };
static const u1_t PROGMEM APPKEY[16] = { 0x2B, 0x7E, 0x15, 0x16, ... };

void os_getArtEui (u1_t* buf) { memcpy_P(buf, APPEUI, 8); }
void os_getDevEui (u1_t* buf) { memcpy_P(buf, DEVEUI, 8); }
void os_getDevKey (u1_t* buf) { memcpy_P(buf, APPKEY, 16); }

void do_send(osjob_t* j) {
  if (LMIC.opmode & OP_TXRXPEND) return;
  int soil = analogRead(34);
  byte payload[2] = { lowByte(soil), highByte(soil) };
  LMIC_setTxData2(1, payload, sizeof(payload), 0);
}

void onEvent(ev_t ev) { /* Event-Handling */ }

void setup() {
  os_init();
  LMIC_reset();
  LMIC_startJoining(); // OTAA-Beitritt
}
```

### 📡 5G (mMTC) – Echtzeit für massive IoT

**Vorteile:**

- Echtzeit-Latenz <10 ms
- Hohe Datenrate (bis 10 Gbps)
- Skalierbar (>100.000 Geräte pro km²)

**Nachteile:**

- Hohe Modulkosten (50–200 €)
- Stromverbrauch zu hoch für reine Batteriegeräte

**Einsatzgebiete:**

- Autonome Roboter in der Fertigung
- Vernetztes Fahren (V2X)
- Augmented/Virtual Reality

### 🌐 MQTT – Das IoT-Protokoll

**Vorteile:**

- Extrem geringer Overhead (2–5 Bytes Header)
- Skalierbar (>10.000 Clients pro Broker)
- Echtzeitfähig (Latenz oft <100 ms)

**Nachteile:**

- Keine eingebaute Verschlüsselung – muss über TLS erfolgen

**Typische Anwendungen:**

- Predictive Maintenance
- Smart Home (Sensor → Cloud)
- Industrie 4.0 (Maschinenüberwachung)

**Analogie:**

- MQTT = **Radio** (ein Sender → viele Empfänger)
- HTTP = **Telefonat** (1:1)

## 3. MQTT-Architektur: Wie funktioniert das Protokoll?


```
[Publisher (Sensor)] --> (MQTT) --> [Broker] --> (MQTT) --> [Subscriber (Dashboard)]
```

| Komponente | Funktion                                   | Beispiele                     |
|------------|--------------------------------------------|-------------------------------|
| Publisher  | Sendet Daten an den Broker                 | ESP32, Raspberry Pi, Sensor  |
| Broker     | Verteilt Nachrichten an Subscriber         | Mosquitto, EMQX, HiveMQ      |
| Subscriber | Empfängt Daten vom Broker                  | Grafana, Cloud-Dienste, Node-RED |

**MQTT-Topic-Aufbau:**  
`<domain>/<device>/<sensor>/<data>`  
Beispiel: `factory/machine1/vibration`

**Wildcards:**

- `+` (Single-Level): `factory/+/vibration` – Daten aller Maschinen
- `#` (Multi-Level): `factory/#` – alle Topics unterhalb von `factory`

## 4. QoS-Level: Zuverlässigkeit vs. Performance

| QoS | Bedeutung                      | Garantie                         | Beispiel                           |
|-----|--------------------------------|----------------------------------|------------------------------------|
| 0   | At most once (Fire and Forget) | Keine Zustellgarantie            | Raumtemperatur (unkritisch)        |
| 1   | At least once                  | Mindestens einmal, Duplikate möglich | Vibrationswarnung                 |
| 2   | Exactly once                   | Genau einmal, kein Duplikat      | Banktransaktion, SPS-Befehle       |

**Empfehlung für IoT-Systeme:**

- Unkritische Sensordaten: QoS 0
- Warnungen/Alarme: QoS 1
- Kritische Steuerbefehle oder Datenbankeinträge: QoS 2

## 6. MQTT-Sicherheit: TLS, Authentifizierung, MQTT 5.0

### 🔒 Transportverschlüsselung (TLS)

**Zertifikate erstellen:**

```bash
openssl req -x509 -newkey rsa:4096 -keyout server.key -out server.crt -days 365 -nodes
```

**Broker mit TLS starten (Mosquitto):**

```bash
mosquitto -v -p 8883 --cafile ca.crt --certfile server.crt --keyfile server.key
```

**Client mit TLS verbinden (Arduino/PubSubClient):**

```cpp
WiFiClientSecure wifiClient;
wifiClient.setCACert(root_ca);
PubSubClient client(wifiClient);
client.setServer("broker-ip", 8883);
client.connect("ClientID", "user", "password");
```

### 🔒 Benutzerauthentifizierung (ACLs) – Mosquitto

`/etc/mosquitto/mosquitto.conf`:

```
allow_anonymous false
password_file /etc/mosquitto/passwd
acl_file /etc/mosquitto/acl.conf
```

`/etc/mosquitto/acl.conf`:

```
user esp32_user
topic readwrite home/+
topic read factory/machine1/vibration
```

### 🔒 MQTT 5.0 – erweiterte Sicherheit & Features

- **Shared Subscriptions** für Lastverteilung
- **Request/Response**-Muster (bidirektional)
- **User Properties** für Metadaten

**Beispiel mit ESP-IDF (esp_mqtt, MQTT 5.0):**

```cpp
esp_mqtt_client_config_t cfg = { ... };
cfg.protocol_ver = MQTT_PROTOCOL_V_5;
cfg.user_property = { { "device_id", "ESP32-001" } };
client = esp_mqtt_client_init(&cfg);
```

## 7. Praktische Anleitungen: Broker einrichten

### 📌 Mosquitto (lokal)

**Installation & Test:**

```bash
sudo apt update && sudo apt install mosquitto mosquitto-clients
sudo systemctl enable --now mosquitto

# Test
mosquitto_pub -h localhost -t "test/topic" -m "Hello MQTT!"
mosquitto_sub -h localhost -t "test/topic" -v
```

**TLS-Konfiguration:**

```
listener 8883
cafile /etc/mosquitto/certs/ca.crt
certfile /etc/mosquitto/certs/server.crt
keyfile /etc/mosquitto/certs/server.key
```

### 📌 EMQX (skalierbar)

**Docker-Installation:**

```bash
docker run -d --name emqx \
  -p 1883:1883 -p 8083:8083 -p 8084:8084 -p 8883:8883 -p 18083:18083 \
  emqx/emqx:latest
```

**Dashboard:** `http://<IP>:18083` (Standard: admin/public)

**Rule Engine für InfluxDB:**

```sql
SELECT payload.vibration AS vibration, clientid() AS device_id, timestamp AS ts
FROM "factory/machine1/vibration"
```

Aktion: `data_to_influxdb` mit URL `http://influxdb:8086`

### 📌 AWS IoT Core (Cloud)

1. **Thing erstellen** (AWS IoT → Manage → Things)
2. **Zertifikat generieren** und Policy anfügen
3. **Policy-Beispiel (Least Privilege):**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["iot:Connect"],
      "Resource": "arn:aws:iot:eu-central-1:123456789012:client/ESP32-*"
    },
    {
      "Effect": "Allow",
      "Action": ["iot:Publish"],
      "Resource": "arn:aws:iot:eu-central-1:123456789012:topic/factory/+/vibration"
    }
  ]
}
```

**ESP32 verbinden:**

```cpp
wifiClient.setCACert(aws_root_ca);
wifiClient.setCertificate(aws_cert);
wifiClient.setPrivateKey(aws_private_key);
client.setServer("a1b2c3d4e5f6g7h8-ats.iot.eu-central-1.amazonaws.com", 8883);
client.connect("ESP32-Vibration");
```

## 8. Fehlerbehebung: 5 häufige MQTT-Probleme

| Fehler                        | Mögliche Ursache                                    | Lösung                                                                 |
|-------------------------------|-----------------------------------------------------|------------------------------------------------------------------------|
| **Broker nicht erreichbar**   | Falsche IP/Port, Firewall blockt                    | `ping`, `telnet` testen, Port 1883/8883 in Firewall freigeben (`ufw allow 1883`) |
| **Nachrichten gehen verloren**| QoS 0, instabiles Netzwerk                          | QoS 1 oder 2 nutzen, `client.loop()` in der Hauptschleife aufrufen    |
| **Authentifizierungsfehler**  | Falsches Passwort oder fehlende Zertifikate         | Zugangsdaten prüfen, TLS-Zertifikate vollständig laden                 |
| **Topic wird nicht empfangen**| Groß-/Kleinschreibung nicht beachtet, Wildcards falsch | Topics exakt definieren, `mosquitto_sub -v -t "#"` zum Debuggen       |
| **Hoher Stromverbrauch**      | Keep-Alive-Intervall zu kurz (häufige PINGREQ)       | Keep-Alive erhöhen (z. B. `client.setKeepAlive(300)`), Deep Sleep nutzen |
