# Kubernetes & WebAssembly – Kaffeekollaborations-Tisch-Edition

> Brechen wir Kubernetes und WASM mal auf das Wesentliche herunter – so erklärt, dass man es am Kaffeekollaborations-Tisch sofort versteht.

---

## 1. Kubernetes (K8s) – Der Fuhrpark-Manager

### Die Ausgangslage (Was ihr schon kennt)

Ihr wisst, was Docker macht. Mit Docker packt ihr eine App in einen Container.  
Ihr tippt `docker run app` und es läuft. Super.

### Das Problem

Was passiert im echten Leben (Enterprise)?

- Ihr habt nicht **einen** Container, sondern **50**.
- Die laufen nicht auf eurem Laptop, sondern auf **10 verschiedenen Servern** im Rechenzentrum.
- Um 03:00 Uhr nachts stürzt Server Nr. 4 ab. Die Kunden kommen nicht mehr auf die App.

### Die Lösung (Kubernetes)

Kubernetes ist **nicht** Docker.  
Kubernetes ist der **Chef**, der aufpasst, dass alle Docker-Container ordentlich arbeiten.

- **Die Auto-Analogie:**
  - **Docker** ist das Auto (es bringt dich von A nach B).
  - **Kubernetes** ist der Fuhrpark-Manager einer riesigen Spedition. Er entscheidet: Welcher Lkw fährt wo? Wer kriegt neuen Sprit? Welches Auto wird ersetzt, wenn es eine Panne hat?  
    *(Das eigentliche Fahren überlässt er den Motoren – bei Kubernetes sind das Container-Runtimes wie `containerd` oder `CRI-O`.)*

### Das Minimal-Beispiel (konzeptionell)

Anstatt zu sagen „Starte diesen Container“, sagt man zu Kubernetes via Textdatei:

> „Hey K8s, ich möchte, dass von meiner Web-App immer genau **3 Kopien** laufen. Mir egal auf welchen Servern. Hauptsache 3.“

Wenn jetzt ein Server nachts um 3 Uhr explodiert und eine Kopie stirbt, merkt Kubernetes das, schnappt sich einen funktionierenden Server und startet dort automatisch die 3. Kopie neu (**Self-Healing**).

- **Für Systemintegratoren (SI):** K8s nimmt euch das manuelle Server-Überwachen und IP-Routing ab.
- **Für Anwendungsentwickler (AE):** Ihr müsst euch nicht kümmern, **wo** euer Code läuft, sondern nur, **dass** er im Container läuft.

## 2. WebAssembly (WASM) / WASI – Die Container-Diät

### Die Ausgangslage (Das Problem mit Docker)

Ein Docker-Container ist toll, aber eigentlich ziemlich schwerfällig.  
Warum? Weil in fast jedem Docker-Image ein kleines halbes Betriebssystem (wie Alpine- oder Ubuntu-Linux) mitgeschleppt wird, nur damit oben drauf drei Zeilen Python- oder Node.js-Code laufen. Das verbraucht viel RAM und der Start dauert oft Sekunden.

### Die Idee von WASM

Was wäre, wenn wir den Code (aus Rust, C++, Go oder Python) so kompilieren könnten, dass er ohne eigenes Betriebssystem im Container auskommt? Stattdessen läuft er in einer extrem schlanken, supersicheren **Sandbox** – einer WASM-Runtime.  
Damit diese Sandbox mit dem Host-Betriebssystem reden kann (z. B. auf Dateien zugreifen oder Netzwerkverbindungen öffnen), gibt es **WASI** (WebAssembly System Interface).  
**WASI ist der Stecker, der ein WASM-Modul mit der Außenwelt verbindet – ganz ohne tonnenschweres Linux.**

- **Die Container-Analogie:**
  - **Docker** ist ein *echter Frachtcontainer*. Um ein Buch zu transportieren, verschiffst du eine riesige Stahlbox, in der das Buch liegt.
  - **WASM** ist ein *Briefumschlag*. Nur der Inhalt, perfekt geschützt, ohne tonnenschwere Stahlwand drumherum.

> **Wichtig:** WASM ist kein Allheilmittel. Es kann klassische Container nicht überall ersetzen, aber es ist eine ultraschlanke Alternative für viele Szenarien (Serverless, Edge, Plugins). Übrigens lassen sich WASM-Module auch ganz normal in Kubernetes-Pods ausführen – die beiden Technologien sind also Freunde, keine Feinde.

### Das Minimal-Beispiel (konzeptionell)

| Merkmal | Klassischer Docker-Container | Moderner WASM-Microservice |
|---------|------------------------------|----------------------------|
| **Inhalt** | Linux-Dateisystem + Bibliotheken + deine App | Nur das kompilierte Binary (`app.wasm`) + eine winzige Laufzeitumgebung |
| **Größe** | Oft 100 MB bis 1 GB | Oft nur 2 MB (plus ein paar MB für die Runtime) |
| **Startzeit** | 500 ms bis 5 s | Unter 1 ms (sofort da) |
| **RAM-Verbrauch** | Mindestens 30–50 MB Grundrauschen | Kaum messbar (wenige Kilobytes/Megabytes) |

- **Für Anwendungsentwickler (AE):** Ihr schreibt euren Code (z. B. in Rust oder Go), kompiliert ihn als `.wasm` und er läuft überall identisch – auf Windows, Linux, Mac oder direkt im Webbrowser.
- **Für Systemintegratoren (SI):** Auf demselben Server, auf dem früher nur 10 Docker-Container Platz hatten, könnt ihr jetzt plötzlich 500 WASM-Module gleichzeitig laufen lassen, weil sie kaum Ressourcen fressen – und das bei sicherer Isolation.

## Zusammenfassung für die IHK-Prüfung

Wenn in der Prüfung gefragt wird, worum es hier geht:

- **Docker:** Verpackt die App.
- **Kubernetes:** Verwaltet Tausende dieser Verpackungen auf vielen Servern gleichzeitig (Skalierung & Ausfallsicherheit).
- **WASM (mit WASI):** Eine extrem schlanke Alternative zum klassischen Container – wir holen das Betriebssystem aus dem Image und setzen auf puren, sicheren Code, der in einer minimalen Sandbox läuft. Das spart massiv Startzeit und Ressourcen und kombiniert sich hervorragend mit Kubernetes.

*Merksatz für die Prüfung: Kubernetes sorgt für Ordnung im Container-Fuhrpark, WASM macht aus trägen Stahlboxen schnelle Briefumschläge – und beide können gemeinsam im selben Pod stecken.*
