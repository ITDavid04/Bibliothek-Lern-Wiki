Leitfaden: SSH-Zugriff für GitHub einrichten
(für Umschüler Anwendungsentwicklung / Systemintegration)
Warum SSH statt Passwort?

Wenn du mit GitHub arbeitest, hast du zwei Möglichkeiten zur Authentifizierung: HTTPS (mit Passwort/Token) oder SSH.
SSH (Secure Shell) ist in der professionellen IT der Standard, da es:

    Sicherer ist: Es verwendet kryptografische Schlüsselpaare anstelle von Passwörtern.

    Bequemer ist: Du musst nach der Einrichtung nicht bei jedem git push oder git pull dein Passwort eingeben.

    Professioneller ist: In der Softwareentwicklung ist es die gängige Methode, um sicher mit Remote-Servern zu kommunizieren.

Das Prinzip: Schlüsselpaare

SSH arbeitet mit einem Schlüsselpaar:

    Der Private Key: Bleibt immer auf deinem Rechner. Er ist dein "digitaler Ausweis".

    Der Public Key: Wird auf dem Server (GitHub) hinterlegt. Er dient als "Türschloss", das nur dein privater Schlüssel öffnen kann.

Schritt-für-Schritt-Anleitung

1. Das Schlüsselpaar erzeugen

Wir erstellen einen Schlüssel vom Typ ed25519. Das ist aktuell der empfohlene Standard, da er sehr sicher ist und schnell arbeitet.

Öffne dein Terminal und gib ein:
bash

ssh-keygen -t ed25519 -C "deine.email@beispiel.de"

    Was passiert hier?
    -t ed25519 legt den Algorithmus fest. -C (Comment) dient nur als Label, damit du später in GitHub weißt, welches Gerät das war.

    Eingabeaufforderungen:

        „Enter file in which to save the key“: Einfach Enter drücken (Standardpfad nutzen).

        „Enter passphrase“: Kannst du leer lassen (einfach zweimal Enter), dann ist der Zugriff ohne Passwortabfrage möglich.
        Wichtig: Wenn du ein Passwort vergibst, musst du es bei jeder Git-Aktion (push, pull, clone) erneut eingeben. Für den täglichen Gebrauch auf einem Entwicklungs-Laptop wird das Leerlassen häufig bevorzugt.

2. Den SSH-Agenten starten und Schlüssel laden

Damit dein Rechner den Schlüssel später automatisch findet und nutzen kann, muss der SSH-Agent laufen und deinen privaten Schlüssel kennen. Führe dazu diese beiden Befehle aus:
bash

eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

Das startet die Schlüsselverwaltung im Hintergrund und lädt deinen privaten Schlüssel. Ohne diesen Schritt kommt es sehr oft zu Permission denied-Fehlern.
3. Den Public Key kopieren

Jetzt musst du den öffentlichen Teil deines Schlüssels anzeigen lassen, um ihn bei GitHub bekannt zu machen:
bash

cat ~/.ssh/id_ed25519.pub

Kopiere die gesamte Zeichenfolge, die mit ssh-ed25519 beginnt. Achte darauf, dass es eine einzige durchgehende Zeile ist – keine Zeilenumbrüche!
4. Schlüssel bei GitHub hinterlegen

    Logge dich auf GitHub.com ein.

    Gehe oben rechts auf dein Profilbild → Settings.

    Wähle links in der Seitenleiste: SSH and GPG keys.

    Klicke auf den Button New SSH key.

    Title: Wähle einen aussagekräftigen Namen (z. B. Ausbildung-Laptop-D3001).

    Key: Füge deinen kopierten Schlüssel ein.

    Speichere mit Add SSH key.

5. Verbindung testen

Prüfe nun, ob dein Rechner mit GitHub kommunizieren darf:
bash

ssh -T git@github.com

Beim ersten Mal fragt dich dein Terminal, ob du dem Host vertrauen willst. Antworte mit yes.
GitHub antwortet dir mit einer Begrüßung:
Hi [DeinName]! You've successfully authenticated...

Für die tägliche Nutzung:
Wenn du ab jetzt ein Repository klonst, musst du immer die SSH-URL verwenden (Format: git@github.com:Benutzername/Repository.git).
Auf GitHub findest du sie, indem du unter dem grünen <> Code-Button den Tab SSH auswählst.
Troubleshooting: Häufige Fehlerquellen

    Permission denied (publickey)
    Der wahrscheinlichste Grund: Schritt 2 (Agent und ssh-add) wurde nicht ausgeführt oder der Schlüsselpfad stimmt nicht.
    Lösung: Führe erneut aus:
    bash

    eval "$(ssh-agent -s)"
    ssh-add ~/.ssh/id_ed25519

    Falsche E-Mail
    Die E-Mail-Adresse im ssh-keygen-Befehl muss nicht zwingend mit deinem GitHub-Profil übereinstimmen – das Label dient nur deiner Orientierung. Falls du aber Benachrichtigungen oder Zuordnungen über die E-Mail erwartest, gleiche sie mit deiner GitHub-Primäradresse ab.

    Falsches Format
    Der Public Key muss eine einzige lange Zeile ohne Zeilenumbrüche sein. Kopiere ihn genau so, wie cat ihn ausgibt.

    Alten Schlüssel überschrieben?
    Falls du bereits einen SSH-Key hattest und jetzt einen neuen erzeugt hast, musst du den alten Public Key in GitHub durch den neuen ersetzen (alten Key löschen, neuen hinzufügen).

Pro-Tipp: Dokumentiere in deinem Wiki (oder einer persönlichen Notiz), auf welchen Geräten du SSH-Keys hinterlegt hast. So behältst du den Überblick und kannst gezielt alte Geräte aus deiner GitHub-Konfiguration entfernen.
