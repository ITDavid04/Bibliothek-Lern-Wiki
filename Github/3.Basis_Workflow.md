🔁 Täglicher Git-Workflow: Branches, Pull Requests & Konflikte lösen
1. Warum Branches?

Stell dir den Haupt-Branch (main oder master) wie ein sauberes Endprodukt vor. Jede neue Funktion, jeder Bugfix bekommt einen eigenen Arbeitszweig (= Branch). So bleibt main immer stabil, und du kannst gefahrlos experimentieren.
Typischer Ablauf im Team

    Du startest mit einem aktuellen main.

    Für eine neue Aufgabe erstellst du einen Feature-Branch.

    Du arbeitest, committest und pusht regelmäßig.

    Du erstellst einen Pull Request (PR) auf GitHub, um deinen Code in main zu integrieren.

    Nach Review und Freigabe wird der Branch gemerged – und du beginnst von vorn.

2. Die wichtigsten Befehle im Arbeitsalltag
Aktuellen Stand holen und aufräumen
bash

git checkout main              # Wechsle in den Hauptbranch
git pull                       # Hole die neuesten Änderungen vom Server

Neuen Feature-Branch anlegen und wechseln
bash

git checkout -b feature/mein-feature-name

Namenskonvention: Kleinbuchstaben, keine Leerzeichen, Bindestriche. Beispiele:
feature/login-page, bugfix/navbar-fix, docs/readme-ergaenzt.
Änderungen verfolgen und committen
bash

git status                     # Was ist neu/geändert?
git add .                      # Alle Änderungen vormerken (oder einzelne Dateien: git add datei.txt)
git commit -m "Kurze Beschreibung, was gemacht wurde"

    Tipp: Schreibe Commit-Messages in der Gegenwartsform und auf Deutsch oder Englisch, aber einheitlich.
    Gut: "Login-Button verlinkt" – Schlecht: "hab den button verlinkt".

Regelmäßig den Hauptbranch einholen (Merge-Konflikte vermeiden)

Während du arbeitest, können andere Code in main pushen. Um deinen Branch aktuell zu halten:
bash

git checkout main
git pull
git checkout feature/mein-feature-name
git merge main

So holst du die neuesten Änderungen in deinen Branch und löst mögliche Konflikte früh, nicht erst beim Pull Request.
Fertigen Branch auf GitHub pushen
bash

git push -u origin feature/mein-feature-name

-u setzt den Upstream-Branch, sodass du später nur noch git push tippen musst.
3. Merge-Konflikte verstehen und lösen

Ein Merge-Konflikt entsteht, wenn Git nicht automatisch entscheiden kann, welche Version einer Datei die richtige ist – z. B. wenn zwei Personen die gleiche Zeile geändert haben.
Wie erkenne ich einen Konflikt?

Nach einem git merge oder git pull zeigt Git:
text

CONFLICT (content): Merge conflict in datei.txt
Automatic merge failed; fix conflicts and then commit the result.

In VS Code und anderen Editoren werden die betroffenen Dateien markiert und enthalten sogenannte Konfliktmarker:
text

<<<<<<< HEAD
Deine lokale Änderung
=======
Die Änderung vom Server
>>>>>>> branch-name

Schritt-für-Schritt zur Lösung

    Datei öffnen und den Konfliktbereich suchen.

    Entscheiden: Möchtest du deine Version, die fremde Version oder eine Kombination aus beiden? Lösche die Marker (<<<<<<<, =======, >>>>>>>) und lasse nur den gewünschten Code stehen.

    Datei speichern.

    Gelöste Datei als bereinigt markieren:
    bash

    git add datei.txt

    Merge abschließen:
    bash

    git commit -m "Merge-Konflikt in datei.txt gelöst"

    (Wenn du mitten in einem Merge bist, öffnet sich oft automatisch ein Editor für die Commit-Message – einfach speichern und schließen.)

    Wichtig: Nach einem gelösten Konflikt immer nochmal testen, ob die Anwendung wie erwartet läuft.

4. Der Pull Request (PR) auf GitHub

Sobald dein Branch gepusht ist, erstellst du einen Pull Request:

    Gehe auf GitHub zu deinem Repository.

    Klicke auf Pull requests → New pull request.

    Wähle als base den Branch, in den du mergen willst (meist main) und als compare deinen Feature-Branch.

    Schreibe einen aussagekräftigen Titel und eine kurze Beschreibung, was dein Code tut.

    Ordne Reviewer zu (dein Ausbilder oder Teamkollege) und erstelle den PR.

Nach der Freigabe wird der Branch gemerged. Anschließend kannst du den Feature-Branch lokal und auf GitHub löschen, um Ordnung zu halten.
5. Nützliche Befehle für den Überblick
Befehl	Zweck
git branch	Zeigt alle lokalen Branches an
git branch -a	Zeigt auch Remote-Branches an
git log --oneline --graph --all	Zeigt den Commit-Verlauf als hübsche Zeitleiste
git diff	Zeigt Unterschiede zum letzten Commit
git diff main	Zeigt Unterschiede deines Branches zu main
💡 Erinnerung: Keine Panik vor Konflikten

Merge-Konflikte sind normal und kein Zeichen von Inkompetenz – im Gegenteil: Sie zeigen, dass Git dich vor Datenverlust bewahrt. Mit etwas Übung löst du sie in wenigen Minuten. Und falls du dich wirklich mal verrennst:
bash

git merge --abort

Dieser Befehl bricht den aktuellen Merge-Vorgang ab und stellt den Zustand vor dem Merge wieder her. Danach kannst du in Ruhe neu ansetzen.
