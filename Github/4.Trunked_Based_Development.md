🌱 Trunk-based Development (TBD) – die Alternative zu Feature-Branches
Was ist Trunk-based Development?

Trunk-based Development (TBD) ist eine Arbeitsweise, bei der alle Entwickler direkt auf dem Hauptbranch (dem „Trunk“, meist main oder master) arbeiten. Kurzlebige Branches sind erlaubt – aber nur für wenige Stunden, höchstens einen Tag.

Die Kernphilosophie: Maximale Integration, minimale Abweichung.

Das ist das genaue Gegenteil des klassischen Feature-Branch-Workflows, bei dem ein Branch oft über Tage oder Wochen existiert, bevor er per Pull Request zurückfließt.
Warum TBD? – „If it hurts, do it more often“

Das Zusammenführen von Code (Merge) tut oft weh, wenn man es selten macht: Je länger ein Branch lebt, desto mehr Konflikte entstehen. TBD dreht den Spieß um und macht das Integrieren zur ständigen Routine – so wird der Schmerz gar nicht erst groß.

    Merksatz: „If it hurts, do it more often.“
    Wenn etwas wehtut (Merge-Konflikte), mach es häufiger – dann wird es einfacher.

Die zwei Hauptprinzipien von TBD
1. Sehr häufige Commits (mehrmals täglich)

    Entwickler teilen ihre Arbeit in kleinste, funktionale Häppchen auf.

    Statt ein ganzes Feature an einem Stück zu bauen und erst am Ende zu mergen, wird der Code mehrmals täglich (oft alle paar Stunden) direkt in den Trunk integriert.

    Vorteil: Konflikte werden sofort erkannt und sind winzig – die gefürchtete „Merge-Hölle“ entsteht gar nicht erst.

2. Extrem kurzlebige Branches (Stunden, nicht Tage)

    Wenn überhaupt Branches genutzt werden, dann nur als Short-lived Feature Branches.

    Sie existieren maximal ein bis zwei Tage, oft nur wenige Stunden.

    Langlebige Release-, Test- oder Team-Branches gibt es nicht.

    Vorteil: Kein Entwickler arbeitet lange auf einem veralteten Stand – alle bleiben dicht am aktuellen Code.

Wie bleibt die Produktion trotzdem stabil?

Wenn ständig Code in den Hauptbranch fließt, stellt sich die Frage: Was, wenn ein Feature noch nicht fertig ist oder nicht sofort sichtbar sein soll?

Die Lösung: Feature Toggles (Feature Flags).

Das sind Schalter im Code, mit denen man neue Funktionen ein- oder ausschalten kann. Ein Feature wird bereits in den Trunk gemerged, ist aber „versteckt“, bis es fertig getestet und bewusst aktiviert wird.

    Beispiel: Du baust eine neue Suchfunktion. Der Code dafür liegt schon in main, aber ein Feature-Flag NEUE_SUCHE steht auf false. Für Endnutzer ist nichts sichtbar. Sobald die Funktion bereit ist, wird das Flag auf true gesetzt – ohne neuen Commit und ohne neuen Merge.

TBD vs. Feature-Branch-Workflow – eine Einordnung
Aspekt	Feature-Branch-Workflow (Teil 3)	Trunk-based Development
Branch-Lebensdauer	Tage bis Wochen	Stunden bis max. 1–2 Tage
Integration in main	Über Pull Request nach Fertigstellung	Direkte Commits oder sehr schnelle PRs
Konfliktrisiko	Steigt mit Branch-Dauer	Sehr gering, weil ständig integriert
Typische Absicherung	Code-Review im PR, CI-Checks	Umfassende automatisierte Tests (CI/CD)
Feature-Umschaltung	Fertiges Feature = Merge	Feature Toggles erlauben inaktiven Code im Trunk
Geeignet für	Teams mit weniger strengen CI/CD-Pipelines	Teams mit sehr hoher Automatisierung und Disziplin
Für Umschüler: Was musst du wissen?

Du wirst im ersten Ausbildungsjahr fast immer mit dem Feature-Branch-Workflow arbeiten. TBD ist eine fortgeschrittene Praxis, die du kennen solltest, weil sie in vielen DevOps-Teams und bei großen Plattformen (z. B. Google, Facebook) Standard ist.

Die wichtigste Erkenntnis aus TBD – unabhängig vom Workflow – lautet:

    Integriere deine Änderungen so oft wie möglich in den Hauptstrang.
    Auch wenn du Feature-Branches nutzt: Mache sie nicht länger als nötig und hole dir regelmäßig die Änderungen aus main per git merge main (siehe Teil 3). So vermeidest du die schlimmsten Konflikte und bleibst teamfähig.
